# 项目概览

## 背景

流程配置变更通常涉及多个来源的资料：系统导出的旧版配置表、新版目标配置表、部门在线更新表、任命调整通知、流程变更会议纪要、聊天记录和规则手册。资料来源分散、格式不统一，人工核对时容易遗漏变更依据，也容易混淆正式依据、弱线索和配置上下文。

本项目通过完全虚构的车企/汽车零部件项目开发流程数据，复现一个“流程配置变更复核”demo，用于展示多来源资料解析、RAG 检索、差异分析、证据匹配和轻量 Agent Router 的组合方式。

## 系统流程

1. 使用模拟数据构建原始资料层。
2. 将 CSV、XLSX、Markdown 和可选 PDF 解析为统一 chunk。
3. 使用 sentence-transformers 和 ChromaDB 建立本地向量知识库。
4. 对比旧版系统配置和新版目标配置，生成变更清单。
5. 使用变更清单中的 evidence_query 检索原始资料。
6. 按证据来源类型判断强变更依据、中等变更依据、弱线索、规则依据和配置上下文。
7. 生成复核优先级和业务处理建议。
8. 通过命令行 Agent Router 或 Streamlit 页面提供交互入口。

## 核心设计

- 原始知识库只包含 `data/` 下的资料，不把 `outputs/change_report.csv` 放入知识库。
- 旧配置表和新版配置表只作为配置上下文，不能单独证明变更原因。
- 任命调整通知和正式会议纪要属于强变更依据。
- 部门在线更新表属于中等变更依据，通常需要正式会议纪要或通知确认。
- 聊天记录和口头通知只作为弱线索，不能直接作为正式变更依据。
- 规则文档用于说明判断和复核规则，不能证明某条具体变更一定发生。

## 当前限制

- 当前回答和报告使用规则模板生成，暂未接入大模型 API。
- 当前 RAG 检索只做本地检索链路验证，暂未构建完整评估集。
- 当前 Web 页面是轻量 demo，不包含权限系统和多人协作流。
- 当前文件接入以本地 `data/` 目录为主，暂未支持在线文件上传和增量更新。
